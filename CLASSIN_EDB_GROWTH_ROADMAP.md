# ClassIn EDB 성장 로드맵

## 문서 목적
- 현재 프로젝트를 앞으로 어떤 형태로 키워갈지 판단 기준을 남긴다.
- 로컬 앱 유지, 공유형 웹앱, 완전한 웹서비스 전환 중 어떤 길이 맞는지 상황별로 정리한다.
- 개발 우선순위를 "웹앱화" 자체가 아니라 실제 사용자 가치 기준으로 재정렬한다.

## 현재 상태 요약
- 현재 구조는 이미 `로컬 웹앱형 MVP`에 가깝다.
- 핵심 역할은 다음과 같이 나뉜다.
  - UI: [ui_prototype/index.html](/C:/Projects/Class_project/edb_make/ui_prototype/index.html), [ui_prototype/app.js](/C:/Projects/Class_project/edb_make/ui_prototype/app.js)
  - 로컬 서버: [app_server.py](/C:/Projects/Class_project/edb_make/app_server.py)
  - 파싱/변환 코어: [build_mvp_export.py](/C:/Projects/Class_project/edb_make/build_mvp_export.py), [preprocess.py](/C:/Projects/Class_project/edb_make/preprocess.py), [segment.py](/C:/Projects/Class_project/edb_make/segment.py)
- 즉, 지금 당장 필요한 것은 "처음부터 새 웹앱 만들기"가 아니라 "지금 로컬 제품을 더 잘 쓰이게 만드는 것"이다.

## 핵심 원칙
- 1순위는 파싱 품질이다.
- 2순위는 업로드 후 바로 검수할 수 있는 UX다.
- 3순위가 배포 방식이다.
- 프론트 교체보다 API 경계 정리가 먼저다.
- 사용자가 늘어나기 전까지는 인프라보다 제품성 완성이 더 중요하다.

## 어떤 상황에서 어디로 가야 하나

### 1. 혼자 쓰거나 내부 테스트 단계
추천 방향:
- `로컬 앱 유지`

이유:
- 지금 구조로도 충분히 빠르게 반복 개선할 수 있다.
- 가장 큰 리스크는 웹 배포가 아니라 파싱 정확도, 촬영 이미지 처리, 결과 검수 흐름이다.

이 단계의 우선순위:
- 업로드 퍼스트 UX
- 자동 파싱 안정화
- 문항 검수/순서 조정
- 오류 메시지와 재시도 UX
- 패키징 안정화

### 2. 같은 공간에서 2~5명이 함께 쓰는 단계
추천 방향:
- `공유형 로컬 웹앱`

이유:
- 한 대의 PC 또는 사내 서버에서 앱을 띄우고 같은 네트워크로 접속하면 된다.
- 완전한 SaaS나 계정 시스템 없이도 충분히 운영 가능하다.

이 단계에서 필요한 것:
- 작업 히스토리 저장
- 공용 저장 경로
- 작업 상태 표시
- 충돌 방지를 위한 세션 개념

### 3. 지점/외부 사용자까지 써야 하는 단계
추천 방향:
- `진짜 웹앱화`

이유:
- 여러 장소에서 접속하려면 브라우저만으로 해결되지 않는다.
- 업로드 저장소, 사용자 계정, 작업 큐, 중앙 DB가 필요해진다.

이 단계에서 필요한 것:
- API 서버 분리
- 사용자 인증
- 세션/결과 저장 DB
- 비동기 파싱 작업 처리
- 파일 스토리지

### 4. 오프라인/보안이 매우 중요한 단계
추천 방향:
- `로컬 앱 강화`

이유:
- 시험지, 학원 자료, 민감한 문서 처리에서는 클라우드 업로드 자체가 부담일 수 있다.
- 이 경우 웹앱화보다 로컬 처리와 배포 안정성이 더 중요하다.

이 단계에서 필요한 것:
- 로컬 OCR 우선
- 로컬 저장
- 로그/백업/복구
- 자동 업데이트

### 5. 모바일 촬영이 핵심인 단계
추천 방향:
- `웹앱화보다 모바일 업로드 UX 강화`

이유:
- 사용자가 원하는 것은 "예쁘게 배포된 웹앱"이 아니라 "휴대폰으로 찍고 바로 처리되는 경험"이다.
- 현재도 모바일 브라우저 업로드 흐름을 개선하면 상당 부분 해결 가능하다.

이 단계에서 필요한 것:
- 카메라 촬영 진입 단순화
- 여러 장 순서 유지
- 업로드 진행 상태
- 재촬영/제거
- 세로 긴 사진 보정 품질

## 지금 시점의 권장 경로
- 현재 시점에서는 `로컬 앱 유지 + 구조 분리`가 가장 맞다.
- 즉시 해야 할 일은 웹 프레임워크 갈아타기가 아니라 아래 두 가지다.
  - 사용자 플로우 단순화
  - API/서비스 경계 정리

## 단계별 실행 로드맵

### Phase 1. 로컬 제품화
목표:
- 지금 EXE 패키지 기반 앱을 실제 사용 가능한 수준으로 안정화한다.

주요 작업:
- 업로드 즉시 자동 파싱 흐름 정리
- 결과 화면 단순화
- 실패 원인 메시지 정리
- 최근 작업 복원
- 샘플/개발용 버튼 고급 설정으로 이동
- 촬영 이미지 품질 향상

완료 기준:
- 비개발자도 `업로드 -> 자동 파싱 -> 검수 -> EDB 생성`을 이해할 수 있다.
- 파싱 실패 시 어디서 왜 실패했는지 알 수 있다.

주요 코드 접점:
- [ui_prototype/index.html](/C:/Projects/Class_project/edb_make/ui_prototype/index.html)
- [ui_prototype/app.js](/C:/Projects/Class_project/edb_make/ui_prototype/app.js)
- [app_server.py](/C:/Projects/Class_project/edb_make/app_server.py)

### Phase 2. 서비스 레이어 분리
목표:
- 웹 전환 여부와 관계없이, 현재 구조를 확장 가능한 형태로 바꾼다.

주요 작업:
- `app_server.py`에서 업로드/응답/파일 접근 로직 분리
- `run_export()` 호출을 서비스 계층으로 이동
- 세션 저장 방식을 파일 JSON 위주에서 SQLite 기반으로 전환
- 파싱 결과를 `세션 -> 업로드 파일 -> 문제 목록 -> 산출물` 구조로 저장

완료 기준:
- UI가 바뀌어도 파싱 코어를 그대로 재사용할 수 있다.
- 한 세션의 입력, 결과, 상태를 일관되게 추적할 수 있다.

권장 신규 구조 예시:
- `services/session_service.py`
- `services/export_service.py`
- `repositories/session_repository.py`
- `models/session.py`

### Phase 3. 작업 기반 API 전환
목표:
- 현재의 동기식 `/api/export`를 장기적으로 더 안정적인 작업 기반 API로 바꾼다.

주요 작업:
- `POST /sessions`
- `POST /sessions/{id}/files`
- `POST /sessions/{id}/run`
- `GET /sessions/{id}`
- `GET /sessions/{id}/problems`
- `POST /sessions/{id}/export`

완료 기준:
- 업로드 후 긴 파싱 작업도 상태 조회로 관리할 수 있다.
- 나중에 공유형 웹앱으로 가더라도 API를 그대로 재사용할 수 있다.

### Phase 4. 공유형 웹앱
목표:
- 같은 네트워크 또는 내부 서버에서 여러 사용자가 접속할 수 있게 만든다.

주요 작업:
- 서버 프레임워크 표준화
- SQLite 또는 PostgreSQL 도입
- 작업 목록/최근 작업 화면 추가
- 결과 보관 정책 추가

추천 기술:
- FastAPI
- SQLite 시작, 이후 PostgreSQL 확장
- 로컬 파일 저장 또는 NAS 공유 경로

완료 기준:
- 여러 사람이 같은 앱에 접속해도 세션이 섞이지 않는다.
- 결과가 중앙에서 관리된다.

### Phase 5. 완전한 웹서비스
목표:
- 지점/외부 사용자, 계정, 중앙 관리가 가능한 서비스형 제품으로 확장한다.

주요 작업:
- 프론트엔드 별도 앱 분리
- 사용자 인증
- 업로드 스토리지
- 비동기 워커
- 중앙 로그/모니터링

추천 기술:
- 프론트: React 또는 Next.js
- 백엔드: FastAPI
- 작업 큐: Celery, RQ, Dramatiq 중 택1
- 저장소: PostgreSQL + S3/MinIO

완료 기준:
- 여러 지역/여러 사용자 환경에서 안정적으로 동작한다.

## 전환 기준

### 로컬 앱에서 공유형 웹앱으로 넘어갈 시점
- 한 사람이 아닌 여러 사용자가 동시에 쓰기 시작할 때
- 결과를 한 곳에 모아 관리해야 할 때
- 최근 작업, 이력, 공유가 필요할 때

### 공유형 웹앱에서 완전한 웹서비스로 넘어갈 시점
- 다른 네트워크/다른 지점에서 접속해야 할 때
- 계정/권한 체계가 필요할 때
- 서버 자원으로 OCR/파싱을 돌리는 편이 더 효율적일 때

### 아직 웹앱화가 불필요한 신호
- 사용자가 1명 또는 극소수일 때
- 가장 큰 문제점이 배포보다 파싱 품질일 때
- 로컬 환경만으로도 충분히 운영 가능한 상황일 때

## 데이터 모델 초안
- `users`
- `sessions`
- `uploads`
- `jobs`
- `problems`
- `exports`

최소 의미:
- `sessions`: 한 번의 업로드/파싱 단위
- `uploads`: 세션에 포함된 원본 파일들
- `jobs`: 파싱/내보내기 작업 상태
- `problems`: 자동 분리된 문항 목록
- `exports`: EDB, 렌더 이미지, JSON 결과물

## 다음 액션 우선순위
1. 로컬 앱 UX를 업로드 퍼스트로 유지하면서 더 간단하게 다듬기
2. 세션/작업 상태를 파일 기반에서 구조화된 저장 방식으로 옮기기
3. `run_export()` 주변의 서비스 레이어 분리
4. 작업 기반 API 설계 시작
5. 필요할 때만 공유형 웹앱으로 확장

## 최종 결론
- 지금 당장은 `웹앱화 자체`가 목표가 아니다.
- 현재 가장 맞는 방향은 `로컬 앱 고도화 -> API 경계 정리 -> 공유형 웹앱 여부 판단` 순서다.
- 즉, 지금 해야 할 일은 새 프론트 스택 도입이 아니라 현재 제품을 실제로 더 잘 쓰이게 만드는 것이다.
